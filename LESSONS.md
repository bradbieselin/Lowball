# Lowball — LESSONS.md
# Master list of bugs, fixes, and patterns. READ THIS BEFORE MAKING CHANGES.

## Expo / React Native

### Use expo-file-system/legacy for file reading
- The new `File` class API (`expo-file-system`) has `arrayBuffer()` but no direct base64 method
- `arrayBufferToBase64()` via string concatenation is CATASTROPHICALLY slow on phones (freezes JS thread on 1MB+ images)
- Use `import * as FileSystem from 'expo-file-system/legacy'` with `FileSystem.readAsStringAsync(uri, { encoding: FileSystem.EncodingType.Base64 })` — it works, the deprecation warning is harmless

### CameraView children don't receive touches in Expo Go
- Placing `TouchableOpacity` inside `<CameraView>...</CameraView>` causes buttons to be unresponsive
- Fix: Render camera overlays (top bar, bottom bar, crosshair) OUTSIDE `CameraView` with `position: 'absolute'` and `zIndex: 10`

### expo-camera plugin config required in app.json
- Must add `expo-camera` and `expo-image-picker` to `plugins` array with permission strings, otherwise camera permissions fail silently on build

### Expo Go version compatibility
- Expo Go app on phones may lag behind the latest SDK. Check compatibility before upgrading. SDK 54 is the current safe target for Expo Go.

### Environment variables are baked at bundle time
- `EXPO_PUBLIC_*` vars are read when Metro bundles, not at runtime
- After changing `.env`, restart Expo with `--clear` flag

## Navigation

### Onboarding → Auth flow: don't use navigation.reset to screens that don't exist
- After onboarding, the user isn't authenticated yet, so `Home` screen doesn't exist in the navigator
- Fix: Set the onboarding flag in state (via a shared setter from RootNavigator), which triggers re-render removing Onboarding screen and showing Auth naturally
- Never `navigation.reset()` to a screen name that isn't currently in the navigator stack

### Double-tap prevention on navigation buttons
- Scan button, deal cards, etc. can fire twice if tapped rapidly
- Fix: Use a `useRef` boolean flag with a 1-second cooldown timeout

## API / Networking

### API timeout must be 45+ seconds for scan endpoint
- Scan makes 2 OpenAI API calls (vision + deals) + Supabase image upload
- 15 seconds is not enough. Use 45 seconds minimum.

### Don't await tracking calls before opening URLs
- `trackClick()` was blocking URL opening, causing a noticeable delay
- Fix: Fire-and-forget with `.catch(() => {})`, open URL immediately

### 401 interceptor pattern
- `apiFetch` checks for 401, attempts `supabase.auth.refreshSession()`, retries once
- If refresh fails, calls `onForceSignOut()` callback to clear React auth state
- The callback is registered from `useAuth` hook via `setForceSignOut()` — this bridges the module-level API code to React state without circular deps

## Server / Express

### Supabase reserves "users" table name
- Prisma model `User` must map to `app_users` (not `users`) via `@@map("app_users")`
- Supabase has its own `public.users` table that references `auth.users`

### Auto-create app_users on first API call
- Supabase Auth creates records in `auth.users` but NOT in our `app_users` table
- The auth middleware does a `prisma.user.upsert()` on every request to ensure the record exists
- Without this, all API calls fail for new users

### Express 5 req.params.id is `string | string[]`
- Express 5 changed param types. Must cast: `const id = req.params.id as string`

### Prisma 6 vs 7
- Prisma 7 requires adapters and `prisma.config.ts` — incompatible with direct Supabase Postgres connections
- Use Prisma 6 (`prisma@^6`, `@prisma/client@^6`) for standard `DATABASE_URL` connection strings
- Delete `prisma.config.ts` if it exists (generated by Prisma 7 init)

### dotenv in Express — use path resolution
- `dotenv.config({ path: path.resolve(__dirname, '..', '.env') })` ensures server/.env is found regardless of cwd
- dotenv v17 also auto-injects from cwd `.env`, but explicit path is safer

### All queries must filter by userId
- `GET /api/scan/:id` must use `findFirst({ where: { id, userId } })` not `findUnique({ where: { id } })`
- Save and unsave must verify scan ownership before modifying
- Click tracking must verify scan belongs to user

### Input validation on all POST/PUT endpoints
- Email: regex + max 255 chars
- imageBase64: typeof string + max 5MB
- Pagination: cap limit at 50, floor at 1
- Click tracking: typeof checks on all fields + range bounds on price

### Rate limiting
- POST /api/scan: 10 requests per minute (calls paid OpenAI API)
- POST /api/track/click: 60 per minute

## UI / UX

### Bottom bar touch targets can overlap with FlatList
- A positioned bottom bar can have touches "fall through" to the FlatList behind it
- Fix: Add `zIndex: 10` and `elevation: 10` to the bottom bar

### Keyboard dismissal on auth screens
- Wrap auth screens in `TouchableWithoutFeedback` with `onPress={Keyboard.dismiss}`

### Save button needs optimistic UI
- Toggle state immediately, fire API call, revert on error
- Add success haptic on save

### Deal prices are AI estimates, not real
- GPT-4o generates estimated prices, not live prices
- Always show disclaimer text: "Estimated prices — tap a deal to see the current price"
- Generate retailer SEARCH URLs (e.g. amazon.com/s?k=...) not fake product URLs

### Scan button alignment
- Don't put text labels inside the same container as centered circular elements — they push the circle off-center
- If a label is needed, position it absolutely or remove it

## Security

### Never expose SUPABASE_SERVICE_ROLE_KEY to frontend
- Only `SUPABASE_ANON_KEY` goes in the frontend `.env` (as `EXPO_PUBLIC_SUPABASE_ANON_KEY`)
- Service role key stays in `server/.env` only

### CORS must not be wildcard
- Use `ALLOWED_ORIGINS` env var, not `cors()`

### Supabase Storage bucket must have RLS policies
- `scan-images` bucket: public read, authenticated upload
